# Benchmark methodology

How performance results for this project must be collected and reported. The purpose
of this document is to make the comparison between our validators and the external
baseline **fair**, and to make it hard to publish a misleading number by accident.

It is written before the comparison harness is built (that is issue #26), so that the
harness implements a methodology we agreed on rather than the other way round.

Scope: UTF-16LE only, little-endian hosts.

---

## 1. What is being compared

| Mode | Command | Threads | Reports |
|------|---------|---------|---------|
| `scalar` | `utf16validate <file>` | 1 | `errorCount = N` |
| `parabix_simd_t1` | `utf16validate --simd --thread-num=1 <file>` | 1 | `errorCount = N` |
| `parabix_simd_t2` | `utf16validate --simd --thread-num=2 <file>` | 2 | `errorCount = N` |
| `parabix_simd_t3` | `utf16validate --simd --thread-num=3 <file>` | 3 | `errorCount = N` |
| `parabix_simd_default` | `utf16validate --simd <file>` | Parabix default | `errorCount = N` |
| `simdutf` | `utf16validate_cl <file>` | 1 (single-threaded library) | `valid = true/false` |

`simdutf` is the Clausecker–Lemire baseline (see
[`external/baselines/clausecker_lemire/README.md`](../external/baselines/clausecker_lemire/README.md)).
It is **single-threaded**. Record which kernel it selected (`utf16validate_cl --impl`);
on arm64 it selects the NEON high-byte implementation, the same strategy as our
byte-oriented Parabix kernel.

---

## 2. The output-semantics mismatch (read this before comparing anything)

Our validators and the baseline **answer different questions**:

| Tool | Question | Output | Must it scan the whole buffer? |
|------|----------|--------|-------------------------------|
| Parabix / scalar | *How many* code units are ill-formed? | `errorCount = N` | **Always** — it must count every error |
| simdutf | *Is the buffer well-formed?* | `valid = true/false` (+ index of the **first** ill-formed unit) | **Only on valid input.** On malformed input it may stop at the first error |

Consequences, which the harness must enforce:

- **On valid input the comparison is fair.** Both tools must examine every code unit to
  conclude "well-formed", so they do the same amount of work. **Valid input is the
  primary performance workload.**
- **On malformed input, timings are NOT comparable.** simdutf can exit at the first
  error while our validator keeps counting to the end. A malformed-input "win" for
  simdutf would measure early exit, not validation speed. Do not publish such a
  comparison as a throughput result.
- Malformed input is therefore used for **correctness and accept/reject behaviour**, not
  for cross-tool throughput. If an accept/reject *timing* comparison is ever wanted, both
  sides must be given the same early-exit task — which our validator does not currently
  support — and that must be stated explicitly.
- **Never fabricate an error count for simdutf.** It does not produce one.

Within the Parabix family (scalar vs. SIMD vs. thread counts) all modes report
`errorCount`, so malformed-input timings *are* comparable **among them**.

---

## 3. Fair comparison groups

Each group answers one question. Do not mix them.

**Group A — Does portable SIMD beat scalar?**
`scalar` vs `parabix_simd_t1`. Same tool, same process model, same output semantics,
both single-threaded. This isolates the SIMD kernel. It is the cleanest comparison we
have.

**Group B — How does portable Parabix SIMD compare with a specialized SIMD library?**
`parabix_simd_t1` vs `simdutf`, **on valid input only**. simdutf is single-threaded, so
the Parabix side **must** be pinned to `--thread-num=1`. Comparing simdutf against
multi-threaded Parabix conflates SIMD quality with parallelism and is not a valid
statement about either.
Caveat: this comparison is still not perfectly clean — see §5 (startup) and §6 (I/O).

**Group C — What does multithreading buy us?**
`parabix_simd_t1` vs `t2` / `t3` / `default`. Same binary, same kernel; the only variable
is thread count. Speedup and efficiency are defined in §8.

**Group D — Accept/reject behaviour on malformed input.**
All modes, over the controlled malformed datasets. This is a **correctness/robustness**
check (does every tool reject what it should?), **not** a throughput comparison — see §2.

## 4. Datasets

Generated by `benchmarks/generate_utf16_benchmark.py` (deterministic, seeded).

**Valid** (primary performance workload — every run must report `errorCount = 0` /
`valid = true`):

| Dataset | Why it matters |
|---|---|
| `default` | Original synthetic mix. Kept byte-stable so historical numbers stay comparable. |
| `english_ascii_heavy` | All-BMP, single code units — the easy case |
| `european_accented` | All-BMP, wider code-unit values |
| `south_asian` | All-BMP (Devanagari, Gurmukhi) |
| `cjk` | All-BMP, high code-unit values |
| `emoji_heavy` | **Surrogate-pair dense** — the stress case for a surrogate-pairing validator |
| `mixed_multilingual` | Realistic blend |

Surrogate density is the interesting variable: `emoji_heavy` vs `english_ascii_heavy`
brackets the range. Report at least those two plus `mixed_multilingual`.

**Malformed** (issue #23): patterns `unpaired_high`, `unpaired_low`, `reversed_pair`,
`odd_trailing_byte`, `random_mix`, `clustered_mix`, at error rates
`0, 0.0001, 0.001, 0.01, 0.1, 1` %. Each file's sidecar carries an
`expected_error_count` computed from the bytes by an independent reference. Use these
for Group D only.

Generated data is git-ignored and must never be committed.

## 5. Input sizes, and why small files lie

- **Preliminary:** 1, 8, 32, 64 MiB.
- **Final:** 128, 256, 512 MiB.

Small inputs are dominated by **fixed per-process cost**, not by validation. This is not
hypothetical — it is visible in our own committed results, where the *same scalar
validator* measures:

- 1 MiB → **46.5 MiB/s**
- 64 MiB → **1354.8 MiB/s**

a ~29× apparent difference that is almost entirely the fixed cost being amortized. A
throughput number at 1 MiB is mostly a measurement of process startup.

**This is the biggest fairness threat in the whole project**, because the fixed cost is
**not the same for the two tools**:

- Parabix pays process startup **plus** loading/JIT-compiling its pipeline (warmups
  populate the on-disk object cache, so measured runs load rather than recompile — but
  the load is still not free).
- simdutf is an ahead-of-time compiled binary with no pipeline to construct, so its
  fixed cost is much smaller.

Comparing whole-process wall-clock at small sizes therefore **systematically favours
simdutf** for reasons that have nothing to do with SIMD quality.

Required mitigations for issue #26:

1. **Headline cross-tool numbers must come from the large sizes** (128–512 MiB), where
   the fixed cost is amortized.
2. **Measure each tool's fixed cost** (e.g. run it on an empty/tiny input) and record it
   as `fixed_overhead_seconds`. Report **both** raw throughput and overhead-adjusted
   throughput, and say which is which.
3. Prefer a harness that can time **steady-state validation in-process**, separately from
   cold start. Record which regime a row belongs to via `timing_scope`
   (`whole_process` | `steady_state`).
4. Never present a small-input cross-tool ratio as a performance conclusion.

## 6. I/O must be equalized

The two binaries do not currently read input the same way: our simdutf wrapper reads the
whole file into memory before validating, and the Parabix tool has its own source kernel.
File-reading cost is therefore inside the timed region for both, but not necessarily in
the same proportion.

Issue #26 must either (a) exclude file reading from the timed region for both tools, or
(b) include it for both and warm the OS page cache identically (the warmup runs already
help). State which was chosen. Do not compare a memory-resident buffer against a
cold-file read.

---

## 7. Measurement rules

- **Warmups:** 2 per configuration (minimum). These prime the OS page cache and the
  Parabix object cache. Warmup timings are discarded.
- **Measured repetitions:** 7 per configuration (minimum); more for the final numbers.
- **Primary statistic: the median.** Also record **min**, **mean**, and **population
  standard deviation**, and keep **every individual repetition** so variance can be
  audited.
- **Throughput** = `MiB / median_seconds`. Derive it from the median, never from the mean.
- **Speedup** = `baseline_median / configuration_median`, with the baseline stated
  explicitly (scalar for Group A/C; `parabix_simd_t1` for Group B).
- **Correctness gate:** every timed run must produce the *expected* result
  (`errorCount` equal to the dataset's `expected_error_count`, or `valid` matching).
  A run that produces the wrong answer is discarded, not timed.
- Prefer a quiet machine (no other heavy load); note if that could not be guaranteed.
- Report **variability**, not just the median. If the standard deviation is a large
  fraction of the median, say so rather than quoting a single number.

**Environment to record with every result set** (otherwise the numbers are not
reproducible):

- CPU model, core count, RAM; OS and version
- Compiler and version; **LLVM version**
- **Parabix commit** (currently pinned: `f0369dd138e2e7a710566d5035f68b9cdc0bf305`)
- **simdutf tag/commit** (currently pinned: `v9.0.0` / `ca7acbcea967b5dcbab490066e99e3a6e6925539`)
- **simdutf selected implementation** (`utf16validate_cl --impl`; e.g. `arm64`)
- Thread count for each Parabix row; dataset type, size, and (if malformed) pattern and
  error rate
- Build type (Release) and any relevant flags

## 8. Multithreading methodology

Thread counts to test: `--thread-num=1`, `2`, `3`, and the Parabix default.

- **Speedup(n)** = `median(t1) / median(tn)` — measured against our **own** single-thread
  SIMD run, not against scalar and not against simdutf.
- **Thread efficiency(n)** = `Speedup(n) / n`. An efficiency well below 1 is the normal,
  expected outcome and should be reported plainly rather than hidden.

Interpretation guidance (state these as *candidate explanations*, and only assert one if
the data supports it):

- **Memory bandwidth.** Validation is a streaming, low-arithmetic-intensity scan. It
  saturates memory bandwidth quickly, so throughput may stop scaling with threads well
  before the core count is reached. This is the most likely limiter here.
- **Scheduling / pipeline overhead.** Parabix's segment threading has coordination cost;
  on small inputs this can make more threads *slower*.
- **Small-file overhead.** At 1–8 MiB the fixed cost (§5) can swamp any parallel gain.
  Thread-count conclusions must be drawn from the large sizes.
- **Kernel structure.** Our validator carries a sequential `pendingHigh` carry across
  blocks/segments. It is handled correctly, but it is a serial dependency in an otherwise
  parallel scan and may limit scaling.

Do **not** compare multi-threaded Parabix against single-threaded simdutf and present it
as a SIMD result (§3, Group B).

## 9. Reporting outputs (contract for issue #26)

### CSV schema

Extend the current schema (`timestamp, host, platform, machine, processor,
python_version, binary, dataset, bytes, mib, mode, thread_count, warmups, repetitions,
min_seconds, median_seconds, mean_seconds, stdev_seconds, throughput_mib_s,
speedup_vs_scalar, individual_times_seconds`) with the columns needed to make a
cross-tool row self-describing:

| Column | Meaning |
|---|---|
| `tool` | `scalar` \| `parabix_simd` \| `simdutf` |
| `parabix_commit`, `llvm_version` | provenance of our binary |
| `simdutf_commit`, `simdutf_impl` | provenance + selected kernel of the baseline |
| `dataset_type` | `default`, `cjk`, `emoji_heavy`, … |
| `error_pattern`, `error_rate_percent` | empty for valid datasets |
| `expected_result` | expected `errorCount`, or `valid` |
| `reported_result` | what the tool actually printed |
| `result_ok` | did it match? (a `false` row must not be used for timing) |
| `timing_scope` | `whole_process` \| `steady_state` |
| `fixed_overhead_seconds` | measured constant cost for this tool on this host |
| `throughput_adjusted_mib_s` | throughput after subtracting fixed overhead |
| `comparison_group` | `A` \| `B` \| `C` \| `D` (§3) |

Keep one row per (tool, mode, dataset, size), with every repetition retained in
`individual_times_seconds`.

### Markdown summary

Machine/provenance block; one table per comparison group (§3); explicit statement of
`timing_scope`; variability alongside every median. Conservative wording only — no
speedup claim that the recorded data does not support.

### Graph inputs

Derived from the CSV, not hand-entered:

- Throughput vs. input size, one line per mode (shows the amortization curve of §5)
- Throughput vs. dataset (surrogate density: `english_ascii_heavy` → `emoji_heavy`)
- Speedup and thread efficiency vs. thread count (§8)
- Group B bar chart: `parabix_simd_t1` vs `simdutf`, **large sizes, valid input only**

---

## 10. Rules for claims

- No performance claim without a recorded, reproducible measurement behind it.
- State the comparison group and the baseline for every speedup number.
- Cross-tool claims come from **valid input at large sizes**, single-threaded on both
  sides, with the startup caveat stated.
- Report negative and flat results honestly (e.g. if multithreading does not help, or if
  simdutf is faster). A truthful null result is a legitimate finding.
- Results are machine-specific until reproduced on another machine.
